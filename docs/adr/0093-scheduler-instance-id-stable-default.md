# ADR-0093: Stable Default for Scheduler `instance_id`

**Status:** Implemented
**Date:** 2026-05-17
**Author:** Leonardo Merza

## Context

### Background

The scheduler app (introduced in [ADR-0024](0024-in-process-task-scheduler.md), hardened in [ADR-0062](0062-scheduler-resilience-improvements.md)) writes per-task health and run history to two tables:

- `SchedulerTaskHealth` — latest-snapshot rollup the UI's "Last run / Duration / Fails / Last error" cells read from (`backend/scheduler/views.py:47-207`).
- `SchedulerTaskRun` — append-only history for the run-details panel.

Both tables key each row by `(task_name, instance_id)`. `SchedulerTaskHealth` has a unique constraint on that pair (`backend/scheduler/migrations/0001_initial.py:79-84`), so each (task, instance) is a single upserted row.

`instance_id` is generated by `scheduler.telemetry.get_instance_id()`:

```python
# backend/scheduler/telemetry.py:21-34
hostname = socket.gethostname()
pid = os.getpid()
_CACHED_INSTANCE_ID = f"{hostname}:{pid}"
```

with an optional override path: if `SCHEDULER_INSTANCE_ID` is set in Django settings, that wins instead.

### Current State

In a default Docker run, `socket.gethostname()` returns the container's randomly-assigned 12-char ID (e.g. `a3f5b8c91234`). Every `docker compose up --build` produces a brand-new hostname, and the supervisor-launched daphne process gets a new PID inside the new container. The composite is fresh per rebuild.

Result: after a rebuild, the scheduler writes its run telemetry under a *new* `instance_id`, while the UI's status endpoint filters by the same new `instance_id` (`backend/scheduler/views.py:53,59`). Both writer and reader see a clean slate — but the prior container's rows are still in Postgres, just orphaned to a hostname that no longer exists.

The UI surfaces the prior rows only via the "orphaned tasks" branch at `views.py:152-198`, which folds in rows whose `task_name` is no longer registered. Rows whose `task_name` *is* still registered but under a stale `instance_id` are invisible in both branches — they get neither folded into the live task row nor surfaced as orphans.

User-visible effect: every container rebuild appears to wipe scheduler last-run metrics, even though the data is intact in Postgres (which itself lives in the infra compose at `/media/cubxi/docker/docker-compose.infra.yml` and survives project rebuilds).

### Process Model

Latchpoint's production and dev containers run a **single** daphne process via supervisor:

- `docker/supervisord.prod.conf:15-16` — `daphne -b 127.0.0.1 -p 8000 config.asgi:application` (no `--workers`).
- `docker/supervisord.dev.conf:6-7` — same single-process layout.

The scheduler runs in-process inside that daphne via `scheduler/apps.py`'s `AppConfig.ready()` hook, so the writer (`update_task_health_finished_*`) and the reader (`SchedulerStatusView`) share one PID for the container's lifetime. The `(task_name, instance_id)` composite is doing work for a multi-worker deployment that doesn't exist here.

### Requirements

- Last-run metrics must persist across container rebuilds without operator intervention.
- No new env var introduced. The existing `SCHEDULER_INSTANCE_ID` setting (`backend/config/settings.py:176`) stays as an escape hatch for a hypothetical future multi-instance deployment but is not required for the single-container path to work.
- Rows already in Postgres from previous container IDs should show up in the UI after the upgrade, not stay orphaned.
- No schema change to `SchedulerTaskHealth.instance_id` (column stays; only its values change).

### Constraints

- `SchedulerTaskHealth` has a unique constraint on `(task_name, instance_id)` — collapsing pre-existing duplicates per task requires deleting losers before updating the survivor's `instance_id`, otherwise the bulk update collides with itself.
- Migration must be idempotent enough to re-run safely if interrupted: deleting non-survivors then updating survivors to a stable `"default"` is naturally idempotent (a second pass finds no losers and a no-op update).
- The migration cannot reconstruct the original `hostname:pid` strings, so the reverse direction is a no-op rather than a raise; raising would block any future schema rollback past this point with no upside.

## Decision

1. **Change the default value returned by `get_instance_id()` from `f"{hostname}:{pid}"` to the constant string `"default"`** (`backend/scheduler/telemetry.py:21-34`). The `SCHEDULER_INSTANCE_ID` override path is preserved unchanged.
2. **Add a data migration** (`backend/scheduler/migrations/0003_consolidate_instance_id.py`) that:
   - For each `task_name` in `SchedulerTaskHealth`, picks the survivor row (most recent by `last_finished_at`, falling back to `last_heartbeat_at` then `updated_at`), deletes the rest, then sets the survivor's `instance_id` to `"default"`.
   - Rewrites every `SchedulerTaskRun.instance_id` to `"default"` (no unique constraint on this table — bulk update is safe).
   - Reverse is a no-op (the prior `hostname:pid` values are intentionally discarded).
3. **Remove the now-unused `import os` and `import socket` from `telemetry.py`**.
4. **Do not change the schema.** The `instance_id` column, indexes, and unique constraint stay. This keeps the door open for a future multi-instance deployment (set `SCHEDULER_INSTANCE_ID` per replica) without a second migration to re-add the column.

## Alternatives Considered

### Pin `SCHEDULER_INSTANCE_ID=latchpoint` in compose

Pros: One-line change in `docker-compose.yml` and `docker-compose.prod.yml`. No code change.
Cons: Adds an env var the user explicitly didn't want to introduce, and existing rows still stay orphaned under their old `hostname:pid` ids until manually rewritten. The fix only works for *future* runs.

### Pin `hostname: latchpoint-backend` in compose

Pros: Stops Docker from inventing a random hostname per container; `get_instance_id()` then returns the same `hostname:pid` across rebuilds *only if PID 1 stays stable*.
Cons: PID still varies in practice (supervisor's process tree isn't pinned), so the fix is partial. And it's the kind of subtle dependency that breaks if anyone touches `supervisord.conf` later.

### Change the read path to aggregate across `instance_id`

Modify `views.py` to return, per `task_name`, the most recent row across *all* `instance_id` values rather than filtering by the current `get_instance_id()`.

Pros: Fixes the symptom without changing instance-id generation. Existing rows immediately visible.
Cons: The writer keeps creating a fresh row per container (the `(task_name, instance_id)` unique key is per-instance), so the table grows unboundedly with one extra row per task per rebuild. Solves the display problem; leaks rows in the schema.

### Drop `instance_id` from the schema entirely

Migrate it away; key everything on just `task_name`.

Pros: Schema honestly reflects the single-instance reality. No latent leak.
Cons: Destructive schema change (column drop + unique-constraint redefinition). Forecloses any future multi-instance deployment without a third migration to add it back. Heaviest blast radius for a benefit the project doesn't currently need.

## Consequences

### Positive

- Scheduler last-run metrics persist across rebuilds. The UI shows real history immediately on first load post-deploy.
- No new env var; deployment surface unchanged.
- Existing orphaned rows are folded back into the live task rows during the data migration, not left as noise.
- The multi-instance escape hatch (`SCHEDULER_INSTANCE_ID`) is preserved untouched — a future operator can opt back into per-instance keying just by setting that env var.

### Negative / Risks

| Risk | Mitigation |
|---|---|
| The migration collapses multiple historical rows per task into one survivor, losing the historical "which container ran this" attribution. | Acceptable: that attribution is meaningless in a single-instance deployment and was never surfaced in the UI anyway. The `SchedulerTaskRun` history table keeps the per-row attribution (just rewritten to `"default"`), so timeline reconstruction is unaffected. |
| If anyone has run `SCHEDULER_INSTANCE_ID=foo` in production at some point, their rows under `instance_id="foo"` get rewritten to `"default"` by the migration. | The migration is intentional: collapsing all instance ids to the stable default is the goal. An operator who wants to *keep* multi-instance keying should set `SCHEDULER_INSTANCE_ID` again after the migration; the survivor row stays correct because the override beats the new default at next write. |
| Future operator wanting horizontal scale must remember to set `SCHEDULER_INSTANCE_ID` per replica, or all replicas will share a single `(task_name, "default")` row and clobber each other's writes. | Documented in `.env.example:66` already (the override hint exists). A `# Required when running multiple instances` comment is added on that line in this change. |

### Neutral

- The `instance_id` column, indexes, and unique constraint remain. No schema-level change; the field's domain just collapses to `{"default"}` in single-instance deployments.
- The `SchedulerStatusView` orphan-tasks branch (`views.py:152-198`) keeps working unchanged — it now sees fewer orphans (zero, ideally, in steady state).

## Implementation Plan

1. Edit `backend/scheduler/telemetry.py`:
   - Remove `import os` and `import socket`.
   - Replace the `hostname:pid` default in `get_instance_id()` with `_CACHED_INSTANCE_ID = "default"`.
2. Create `backend/scheduler/migrations/0003_consolidate_instance_id.py`:
   - `RunPython` operation that collapses health rows and rewrites run-history rows as described above.
   - Reverse is a no-op function.
3. Update the comment on `.env.example:66` from `# Auto-generated if empty` to `# Defaults to "default"; set per-replica only for multi-instance deployments.`
4. Run `uvx ruff check backend/` + `uvx ruff format --check backend/` and the scheduler test app via `./scripts/docker-test.sh`.
